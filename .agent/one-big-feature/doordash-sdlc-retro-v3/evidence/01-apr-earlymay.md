## Project shape (for context)
- **Product:** "SEIU 521 DoorDash Linker v2.0" — an email-inbound automation. Union members email required documents to an inbound address; the system ingests attachments, classifies them, matches DoorDash orders to meeting approvals, and emails failure/success notices.
- **The 3 required documents** (the compliance core): (1) meeting approval form, (2) meeting sign-in sheet (scanned), (3) DoorDash order confirmation.
- **Stack:** TypeScript, Trigger.dev (job orchestration), Supabase (Postgres + storage), Resend (inbound + outbound email), Anthropic (multimodal extraction), Egnyte (source doc store, reused from v1). Drizzle ORM.
- **v1 → v2:** v1 was Clearworks.ai-owned infra (Supabase `zdkfqoxqwnxplerciyjq`… note: reused name; two projects exist). v2 = migrate everything to SEIU-521-owned infra.

## Key scope-creep / unpriced-discovery moments (the ones that matter)

> **The through-line:** the same assumption broke twice. The original design keyed off *counting/naming files* (exactly-3 attachments; classify by filename). Real submissions don't respect that — members merge docs, and filenames lie. Both breaks forced the system to *read the actual content* instead. The retro should treat "count/filename shortcut → must read content" as one recurring pattern, not two isolated bugs.

0. **Combined-attachments: exactly-3 gate → content-coverage validation (Josh 2026-04-21, worked 2026-04-22).** The *earliest and most structural* scope break, and the one the prior report most likely compressed away. PRD v2 §4.3 hard-failed any submission where `attachment_count != 3`. Josh, from the field: *"sometimes… the users might combine the sign in seert with the approval form - like we might not have 3 files, though we will have the correct content inside."* The entire happy path assumed 1 file = 1 document type. Fix required dropping the file-count gate, adding a post-extraction content-coverage check (DoorDash fields / approval+signatures / sign-in names), a new `submissions.content_coverage` jsonb column, and PRD rewrites (§4.3 + §4.1.2). Left two open product questions (min file count; block-vs-warn on a missing section — v1 blocked).

1. **Filename heuristics → full vision OCR (2026-05-06).** Same theme, second occurrence. Original doc-classification plan matched filenames + a Haiku fallback. Josh rejected it outright — *"you have to do the full OCR pass and actually figure out which files you have."* Forced a rewrite to Sonnet vision OCR reading actual PDF/image bytes, plus an SDK upgrade (0.30.1→0.94.0) for document-block support, plus a schema change (`received_doc_types`). A "which doc is missing?" feature quietly became a full multimodal compliance-grade OCR classifier.

2. **Failure-email spec drifted across 4+ versions in one session (2026-05-06).** v…506.2 (static 3-attachment copy) → v…506.3 (must reflect *actual* error reason) → v…506.4 (per-doc present/missing) → v…506.5 (OCR-derived). Each iteration was a Josh correction, not a planned increment — classic requirement-discovered-on-contact.

3. **Migration left the pipeline writing to the wrong database (2026-05-07).** The "clean" infra migration silently broke ingestion: `trigger.config.ts` project ref + root `.env` still pointed at the OLD Clearworks Supabase. Submissions were landing in the wrong project. Only surfaced because Josh noticed zero new rows. Unpriced post-migration re-wiring + smoke test.

4. **Client-owned DNS / email account not yet in hand (2026-05-04 → 05-07).** Real inbound address `doordash@dd.seiu521.org` lives in the *SEIU-521* Resend account, not Clearworks'. Josh lacked DNS access at migration time; interim fallback `521@hello.clearworks.ai` had to be used. Created a cross-account visibility gap (can't inspect real submissions via the Clearworks Resend MCP).

5. **Trigger.dev v3→v4 + externalRef traps (2026-05-04).** SDK/CLI version mismatch silently produced "0 tasks detected"; deploy also required `externalRef` not the URL slug. Neither was in the plan — discovered by direct API calls during the migration.

6. **Credential/tooling pacing blockers (2026-04-21).** Repeated multi-hour idle windows where v2 was blocked on a missing `ANTHROPIC_API_KEY`, a Trigger.dev webhook URL, and a send-only-vs-Full-Access Resend key mixup. Also Resend inbound routes are dashboard-only (no API), and newly-wired MCPs don't hot-load into a running session.
