# Fable design — building Josh's self-interview / work-inventory INTO cortext

## What the interview is
Josh's Full Work Inventory is a self-interview rendered as a traffic-light board: every recurring unit of his work, expressed as input→output, tagged with an automation status (already-built-cron / needs-skill / human-only). The grounded pass (08-work-inventory-scaffold.md) upgraded it from self-report to evidence: every green item cites an exact live cron (name + schedule, e.g. `crm/fireflies-ingest` every 2h) and every skill cites a real folder, across 7 categories including two Josh missed (Engineering, Knowledge). Why it matters as a system primitive: it is simultaneously (a) the fleet's job description — the ground truth of what the system is FOR, (b) a routing table — which agent/cron/skill owns each unit of work, (c) a backlog — the blue needs-skill items ARE the roadmap (follow-up drafting, AR scan, LinkedIn publish loop, proposal workflow), and (d) the product — for a client tenant, this exact artifact is the discovery-audit deliverable Josh already sells (Busywork Audit shape), and 'flip your items from red to green' is the retainer. Right now it exists as an HTML/PNG and a markdown file: a snapshot that rots the day a cron changes. Everything below turns it into a living, machine-readable, self-verifying store that the fleet reads, updates, and renders — using only pieces that already run.

## Integrations (compose existing pieces)

### work-inventory.json — the inventory as a bus-native state file
- **What:** Convert 08-work-inventory-scaffold.md into a per-org machine-readable store (e.g. orgs/clearworksai/state/work-inventory.json). Schema per item: id, category, title, input→output, status (live-cron|has-skill|needs-skill|human-only), evidence ({cronAgent, cronName, lastFired, fireCount} or {skillPath}), ownerAgent, lastVerified, seenCount, notes. Statuses are only valid WITH evidence — same discipline as the WS2 claim gate: a green item must cite a cron that actually fired, not a belief.
- **Uses existing:** 08-work-inventory-scaffold.md is the seed content (already cited by exact cron/skill name — it converts mechanically); bus state-file conventions (state/*.txt|json already used by meeting-commitments dedup); the certainty memory's 'ground in CRON-FIRED markers, not fields' rule; WS9's one-canonical-store doctrine applied to work identity instead of entity identity.
- **Value:** One queryable place answering 'what does Josh's work consist of and how automated is each piece' — the prerequisite for every other integration, and the thing agents can consult instead of Josh re-explaining his business every session.

### Inventory reconciler — a nightly cron keeps the board honest automatically
- **What:** A deterministic nightly script-cron (larry or sage) that diffs work-inventory.json against reality: every agent's crons.json (16 agents, all under cortextos1) + ~/.claude/skills/ + org skills. Cron deleted or silent (fire_count stale) → item auto-degrades green→blue with a drift receipt; new cron appears → matching item upgrades or an 'unmapped automation' entry is drafted for Josh; new skill folder → has-skill. lastVerified stamps refresh only from evidence. Drift goes to the WS4 receipt channel, never raw to Josh.
- **Uses existing:** sage/fleet-health-check + sage/nightly-metrics already read the same ground truth every night; WS4 fleet-reconcile + drift alarms is the exact pattern (process/cron/.env diff) extended by one target; larry/context-budget-weekly already proves the 'count the config, alert on change' cron shape; larry/task-reconcile proves auto-flipping status from artifacts (merged PR → task done) — same move, cron → inventory item.
- **Value:** Kills the snapshot-rot problem. The muse constitution config-drift incident (rewrite never made it into crons, found weeks later) is exactly the failure class this closes: the inventory becomes self-verifying instead of another document that lies.

### Blue→green pipeline — needs-skill items feed skill-creator via the improvement dispatch
- **What:** Point frank2/daily-improvement-dispatch (already live, 10am weekdays, gap-only prompts) at work-inventory.json: pick ONE blue item per cycle (current queue is short and concrete: follow-up drafting, AR/invoice scan, LinkedIn selection-to-publish, proposal-drafting workflow, monthly revenue snapshot). If inputs/outputs are already specified on the item, hand straight to /skill-creator to scaffold SKILL.md and (if recurring) a cron; if under-specified, run one Chase-AI-style mini-cycle with Josh over Telegram first ('TASKS I HEARD / COULD BE SKILLS' → confirm → build-gate). On merge, the reconciler — not the builder — flips the item green when it sees the cron fire. Approval via Telegram (or the one-tap decision queue once it lands).
- **Uses existing:** frank2/daily-improvement-dispatch cron (live); /skill-creator (the Chase system's designated hand-off target, never writes files itself); the onboarding-discovery skill design from 06-phase2-capabilities.md; WS2 rule that 'built' claims need a fired receipt.
- **Value:** The inventory stops being a diagnosis and becomes an engine: the fleet works through Josh's own stated automation backlog one item a day/week, with Josh approving from his phone. This is also the demo reel — each flip is a building-in-public content seed muse's fleet-activity-intel cron already captures.

### onboarding-discovery writes the inventory — the interview's native output format
- **What:** Make the planned onboarding-discovery skill (Chase-AI prompt-discovery wire-in) emit work-inventory.json entries as its structured output, in both modes: USER mode refreshes Josh's inventory (each interview cycle's 'TASKS I HEARD' = items, 'COULD BE SKILLS' = status proposals, on-demand/local/cloud tags map to skill-vs-cron); TENANT mode seeds orgs/<client>/state/work-inventory.json with everything status=human-only alongside tenant-identity.json, gated by upstream's ONBOARDING.md — agents can't operate until the client's inventory + identity exist. Add a lightweight quarterly staleness check: items unverified for N weeks in a category trigger a short re-interview cycle over Telegram, so the interview becomes periodic maintenance, not a one-off.
- **Uses existing:** 06-phase2-capabilities.md item 7 (Chase-AI discovery → dual-mode onboarding, Josh-approved); upstream ONBOARDING.md gate + 'bus identity-established' event; the 'client onboarding interview engine' ideation item; Josh's own hard rule 'interview FIRST, external research is input not substitute'; Fireflies capture if the interview happens on a call.
- **Value:** For Josh: the interview that produced the PNG becomes a repeatable loop with a durable output. For clients: discovery day one produces their inventory, the engagement roadmap is literally 'flip these blue', and the generic-analysis failure class (the Marcos framework-overlay mistake) becomes structurally impossible — agents run only on the client's own answers.

### Automation Map tab on briefs — the PNG becomes a live page
- **What:** Render work-inventory.json as a traffic-light board tab on the existing ops dashboard: categories as columns, items with status color, evidence tooltip (cron name, last fired), lastVerified stamp, drift flags from the reconciler, and green/blue/red counts as the headline. Per-tenant later: the same renderer on the client's briefs deployment = a white-label 'what your agency automated for you' status page (composes with the SLO/status-page ideation item and the battle-tested railway-custom-domain skill).
- **Uses existing:** briefs dashboard tab pattern (morning brief IS a dashboard tab per memory); build_dashboard.py + publish-brief.py + the permanent tokened URL; larry/refresh-briefs-dashboard 30m cron already rebuilds tabs; frank2/daily-ops-dashboard publish path; phone-first delivery rule (tappable URL, never file paths).
- **Value:** Josh gets his self-portrait as a living page he can check from his phone; drift is visible the day it happens; and for clients it turns the fleet's internal honesty machinery into a sales artifact — the vendor whose product shows its own receipts.

### Inventory-aware routing + gap detection from live work
- **What:** Two hooks: (1) frank2's dispatch consults the inventory before improvising — a request matching a live-cron/has-skill item routes deterministically to the owning agent/skill (encode as a dispatch-time lookup, same spirit as the Marcos→CRM-only routing rule but generalized and data-driven). (2) The transcript-scanner and meeting-commitments workers (both already sweep sessions/transcripts every 2h) increment seenCount when Josh manually does or requests something matching a human-only/needs-skill item; a threshold (e.g. 3 occurrences) escalates it in the blue queue or drafts a brand-new item for the next interview cycle. Ingest work-inventory.json into the shared MMRAG collection so any agent can semantically answer 'do we already have something for X?' before building.
- **Uses existing:** frank2 dispatch rules pattern (MEMORY routing rules → now data instead of prose); frank2/transcript-scanner + meeting-commitments-worker crons (live, 2h); larry/kb-reconcile-nightly MMRAG pipeline; sage/catalog-browse (Sundays) can match community-catalog skills against open blue items instead of browsing blind.
- **Value:** Closes the loop in both directions: work routes to existing automation instead of being rebuilt ad hoc, and recurring manual toil surfaces itself as the next automation candidate — the fleet notices what Josh keeps doing by hand before Josh has to say it.

## Build plan
Sequence (respects the WS1–WS10 foundation; nothing here blocks it, and steps 1–3 are near-pure composition): STEP 1 — Schema + seed (small, ~half a day): define the item schema, mechanically convert 08-work-inventory-scaffold.md into orgs/clearworksai/state/work-inventory.json (the hard grounding work — citing every cron/skill — is already done in that file). New: one JSON file + a tiny read/validate helper. STEP 2 — Reconciler cron (small): deterministic script (no LLM needed for the diff), scheduled nightly on larry or sage, reading the same crons.json/skills ground truth sage's health crons already read; drift → WS4 receipt channel. New: one script + one cron entry. Reused: fleet-reconcile pattern, task-reconcile's artifact-flips-status move. STEP 3 — Automation Map tab (small): add a renderer to the existing briefs dashboard build; larry/refresh-briefs-dashboard already re-publishes every 30m. New: one tab template. STEP 4 — onboarding-discovery emits inventory (medium): this rides the already-approved Phase 2 item 7 build; the only addition is making work-inventory.json the skill's structured output in both USER and TENANT modes, plus the ONBOARDING.md gate check and a quarterly staleness re-interview trigger. STEP 5 — Blue→green dispatch (small-medium): modify frank2/daily-improvement-dispatch's prompt to draw from the blue queue and hand to /skill-creator with Telegram approval; the reconciler (step 2) does the status flip on evidence, so no new verification machinery. STEP 6 — Routing + seenCount hooks (medium, last): dispatch-time inventory lookup in frank2, seenCount increments in the two existing 2h workers, MMRAG ingest via the existing kb-reconcile path. Total genuinely-new surface: one JSON schema, one diff script, one dashboard tab, and prompt/output-format changes to four things that already run. Everything else is wiring.

## Top picks
- 1. work-inventory.json + the nightly reconciler cron (steps 1–2): the primitive plus its truth-maintenance. Cheapest to build (the grounded content exists in 08-work-inventory-scaffold.md; the reconciler is a diff over files sage already reads nightly) and everything else depends on it. Without the reconciler it's just another rotting doc; with it, the inventory is the first fleet artifact that verifies itself.
- 2. onboarding-discovery emitting the inventory in dual mode (step 4): rides the Phase-2 build Josh already approved, and it's the multiplier — it makes Josh's self-interview repeatable, and it makes every future client tenant start life with their own inventory + identity gate. This is the piece that turns Josh's discovery consulting into the product's setup wizard.
- 3. Automation Map tab on briefs (step 3): near-trivial on the existing dashboard pipeline, phone-first per Josh's delivery rule, and it's the payoff surface — Josh sees his living board daily, drift is visible immediately, and the same renderer becomes the white-label client status page later. (Blue→green dispatch follows naturally once these three exist.)
